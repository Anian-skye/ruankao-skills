## 项目背景编写原则

- 时间控制在考试前 5 年内，且项目最好已完成或阶段性上线。
- 周期、团队规模、金额和系统复杂度要相互匹配。
- 项目名称、公司名和业务数据应脱敏。
- 本人角色必须能支撑论文题目，不要把职责写得过大或过虚。
- 背景段重点交代“业务痛点、建设目标、本人职责”，不是公司宣传。

## 论文段落结构

推荐正文段落使用：

```markdown
理论论点 -> 项目问题 -> 落地措施 -> 效果/反思
```

常见句式：

- “在该项目中，为了解决……问题，我……”
- “由于原有系统……，因此我在架构设计中……”
- “通过上述措施，系统在……方面得到改善。”
- “该实践也暴露出……不足，后续我通过……继续优化。”

避免：

- 纯定义式：“某技术是……”
- 纯口号式：“极大提升效率，效果显著。”
- 纯堆砌式：列一串中间件或方法名，但没有项目动作。

## 常见知识点映射

以下是通用映射方向，必须结合用户自己的项目背景卡改写。

### 架构设计类

- 微服务 / SOA：服务拆分、接口契约、服务治理、集成与扩展。
- 架构风格：分层、事件驱动、管道过滤器、仓储等风格与质量属性匹配。
- 云原生：弹性伸缩、容器化部署、可观测、韧性设计。
- 可观测：日志、指标、链路追踪、告警闭环。
- 分布式事务：一致性边界、补偿、幂等、最终一致。
- 数据集成：多源数据接入、数据质量、血缘、主数据或主题建模。
- 安全架构：认证授权、审计、脱敏、加密、访问控制。

### 软件工程类

- 需求分析：角色识别、痛点澄清、需求建模、原型验证。
- 软件设计：模块划分、接口设计、设计模式、低耦合高内聚。
- 软件测试：单元、集成、系统、性能、安全、自动化测试。
- DevOps：持续集成、质量门禁、灰度发布、回滚。
- 运维：监控告警、容量治理、故障定位、应急预案。
