# ruankao-thesis-argument-prep

本技能把抽象论文知识点落到用户自己的项目背景中。它不内置任何固定项目，也不替用户编造私人经历；必须优先读取用户上传的资料和项目背景卡。

## 前置资料

默认资料布局由 `ruankao-init` 创建：

- 教材资料：`软考/资料/教材/`
- 论文范文：`软考/资料/论文范文/`
- 论文知识：`软考/资料/论文知识/`
- 项目背景卡：`软考/资料/项目背景/项目背景卡.md`
- 论文素材输出：`软考/输出/论文素材/`

如果项目背景卡不存在，先让用户用 `ruankao-init` 创建；也可以在当前轮只输出“项目背景卡待补字段”，不要使用示例项目代替。

## 执行流程

### 1. 识别论点

用户可能只给知识点，例如“微服务架构”“系统测试”“需求分析”。先把它收敛成 1 个可写论点：

- 架构设计：强调架构选择、质量属性、扩展性、治理与演进。
- 案例/综合题迁移：强调采分点、场景判断、设计取舍。
- 论文题材：强调“理论 + 项目动作 + 成效”的正文段落。

如果知识点过大，先拆出 3-5 个常见论点方向，再选最贴合项目背景卡的一项。

### 2. 回查教材角度

先在 `软考/资料/教材/`、`软考/资料/论文知识/`、`软考/资料/论文范文/` 中检索，不凭印象展开。

推荐命令：

```bash
python3 .agents/skills/ruankao-thesis-argument-prep/scripts/search_textbook_angles.py \
  --vault /path/to/vault \
  "SOA" "服务化封装" "集成" "扩展"
```

若脚本找不到目标教材，用 `rg` 或 `pdftotext` 在用户上传资料中检索。只引用短关键词，正文必须转述，不长篇复制教材原文。

### 3. 读取项目背景卡

项目背景卡至少应包含：

- 项目名称或脱敏项目类型
- 建设时间、周期、规模、团队和本人角色
- 业务目标与核心痛点
- 3-5 个可反复复用的业务场景
- 本人负责的架构、设计、治理、测试、运维或交付动作
- 可量化或可信的成效
- 不足与后续改进

写论文素材时只使用卡片中已有事实。若缺字段，使用中层抽象表达，并在输出里标注“需用户补充”。

### 4. 映射到项目实践

每段素材默认采用：

1. 教材角度：说明这个知识点适合从什么方向切入。
2. 理论论点：用 1 句话点明核心主张。
3. 项目问题：从项目背景卡抽取相关痛点或约束。
4. 落地动作：写本人在项目中的设计、组织或实施动作。
5. 成效反思：写可验证成效、经验或不足。

不要只讲故事，也不要只背概念。论点必须先行，项目动作必须服务论点。

### 5. 判断素材等级

按备考价值标注：

- `必背`：高频、可作为主论文题眼，能覆盖多个子题。
- `重点`：适合作为分论点或重要备选题。
- `可辅助`：适合作为补充论据、过程类支撑点或迁移素材。

判断依据优先来自用户上传的考试大纲、教材、真题回忆、范文和个人复习优先级；没有资料时只给保守建议。

### 6. 持久化

用户要求保存时，写入 `软考/输出/论文素材/`：

- 一个知识点优先维护一个笔记。
- 同一知识点的不同论点追加为“素材 01 / 素材 02”。
- 每条素材写明 `等级`、`教材角度`、`项目来源字段`。
- 如果维护索引，使用 `软考/输出/论文素材/知识点索引.md`。

默认不写入，除非用户明确要求“保存、落盘、加入素材库”。

## 输出模板

```markdown
等级：必背 / 重点 / 可辅助

教材切入角度：
……

教材角度论述：
……

论点对应段落：
在该项目中，……

映射说明：
- 项目场景：……
- 落地动作：……
- 体现的知识点：……
- 可写成效：……
- 需补充信息：无 / ……
```

## 禁止事项

- 不要使用任何内置固定项目背景。
- 不要编造用户没有提供的行业、金额、团队规模、业务系统或个人职责。
- 不要把项目实践写成空泛口号。
- 不要脱离题目论点堆砌技术名词。
- 不要默认写入文件；开源版默认让用户确认后落盘。
