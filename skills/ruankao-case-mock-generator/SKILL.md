# ruankao-case-mock-generator

先按用户上传资料归纳软考案例题的出题习惯，再据此生成模拟题，不要凭印象自由发挥。

## 先建立命题上下文

1. 先读 `references/案例出题风格参考.md`，掌握本技能约定的题型、问法、答案口径和禁用写法。
2. 需要快速选题或映射真题类别时，先运行：

```bash
python3 ./.agents/skills/ruankao-case-mock-generator/scripts/pick_case_mock_brief.py \
  --vault /path/to/vault \
  --mode full
```

3. 用户指定章节、方向或题量时，用命令参数覆盖默认值，例如：

```bash
python3 ./.agents/skills/ruankao-case-mock-generator/scripts/pick_case_mock_brief.py \
  --vault /path/to/vault \
  --mode mini \
  --count 3 \
  --focus "UML,Redis,质量属性"
```

4. 如果题库索引不存在，先提示用户运行 `ruankao-init` 并上传案例教材/真题，再运行 `ruankao-case-sprint-10` 的 refresh 生成索引。
5. 如果脚本结果和用户要求冲突，以用户要求为准，但仍保持真题风格。

## 选择输出模式

按用户意图在以下模式中选一种：

- `single`：1 道完整案例题，含背景和 3 到 4 个小问。适合“出一道模拟题”。
- `mini`：2 到 3 道案例题。适合“来一组练习”。
- `full`：默认 5 道整套模拟题，结构贴近用户上传真题或已有练习的组织方式。

如果用户只说“出一套”，默认按 `full` 处理；如果只说“出一题”，默认按 `single` 处理。

## 生成题干时遵守的硬规则

1. 先写业务背景，再列功能、约束、故障或扩展要求。
2. 题干必须场景化，不要直接把知识点名称暴露给用户。
3. 每道大题默认拆成 3 到 4 个小问，优先使用 `（1）（2）（3）` 形式。
4. 小问必须尽量贴近采分点，不写“谈谈理解”“总结常见考点”这类空泛问法。
5. 同一道题里应同时出现：
   - 一个偏识别/补全/判断的小问
   - 一个偏原理/比较/说明的小问
   - 一个偏设计/选型/优化/落地的小问
6. 分值要自然，常见为 `4 / 6 / 8` 分，不要出现很碎的奇怪分值。
7. 用词优先采用真题常见动词：
   - `补充完善`
   - `指出`
   - `说明`
   - `比较`
   - `选择并说明原因`
   - `结合题干分析`
8. 除非用户明确要求，不要把“知识点提示”“答题方向”“考点总结”写进题目本身。

## 常见命题骨架

### 1. 系统设计与建模

优先写成以下结构之一：

- 题干给业务功能，要求补全 `DFD / E-R / UML` 图中的空白、角色、实体、关系或步骤。
- 先让用户识别对象、关系或图元素，再追问图的作用、关系类型或建模原则。
- 最后补一个短说明题，例如“用 100-200 字说明某图或某方法在分析/设计阶段的作用”。

### 2. 系统架构设计与评估

优先写成以下结构之一：

- 从题干中识别质量属性，再写质量属性场景或区分开发期/运行期质量属性。
- 根据多个约束选择架构风格、模式或战术，并把原因落到题干约束。
- 让用户比较两种架构方案或指出风险点、敏感点、权衡点。

### 3. 数据库系统与缓存设计

优先写成以下结构之一：

- 先问原理或差异，如 `RDB vs AOF`、分区 vs 分片、主从 vs Cluster。
- 再问故障分析，如穿透、击穿、雪崩、热点、内存增长、数据丢失。
- 最后问方案设计，要求写出至少 3 种措施并分别说明适用场景。

### 4. Web 应用与综合架构

优先写成以下结构之一：

- 先分析性能恶化、热点冲击、上传超时、链路重复等现象的原因。
- 再要求给出组件职责、数据流转、异步解耦或缓存预热方案。
- 最后补一个比较类小问，如水平拆分/垂直拆分、同步/异步、实时/离线。

### 5. 系统安全

优先写成以下结构之一：

- 从业务合规或数据保护场景切入，要求识别安全目标或安全机制。
- 让用户说明加密、证书、签名、访问控制、审计日志各自作用。
- 最后把安全措施映射到系统架构或业务流程。

## 不像真题的写法

避免以下问法：

- `请口述这个知识点`
- `请总结它的常见考法`
- `请说说有哪些关键词`
- `谈谈你的理解`
- `这个知识点的易错点是什么`
- 只有定义，没有题干场景和业务约束
- 一道题只围绕“定义 + 作用 + 场景”机械展开

如果发现题目太像面试题、太像背书题、太像教学提纲，要重写。

## 落盘要求

每次生成模拟题后，都必须把结果写入 `软考/输出/测试/` 目录，不要只在对话里给出。

必须同时产出两个独立文档：

- 题目文档：`软考/输出/测试/案例模拟测试-YYYY-MM-DD[-主题].md`
- 参考答案文档：`软考/输出/测试/案例模拟测试-YYYY-MM-DD[-主题]-标准答案.md`

不要把题目和答案写在同一个文件里。

题目文档至少应包含：

1. frontmatter
2. 测试标题、来源说明、建议用时
3. 每道案例题的题干和小问
4. 指向参考答案文档的链接

参考答案文档至少应包含：

1. frontmatter
2. 对应题目文档的反向链接
3. 按题号和小问展开的标准答案
4. 采分点或推荐答案组织

建议沿用用户工作区已有测试文件的 frontmatter 和链接方式；若没有现成样式，使用最小 frontmatter：`title/date/tags/standard_answer/status`。

## 写标准答案的方式

生成模拟题时，默认同时生成并保存参考答案；即使用户主要想要题目，也仍应配套产出另一份答案文档。

标准答案必须满足：

1. 按 `题目一 -> （1） -> 采分点` 组织。
2. 每个小问优先拆成 2 到 6 个采分点，不要只写一段大白话。
3. 允许有“推荐答案”，但必须以教材口径和真题常见说法为主。
4. 如果是选型题，答案必须把“为什么适合该场景”写出来。
5. 如果是比较题，答案必须把对比维度写清楚。
6. 如果是质量属性题，答案必须尽量落到 `刺激源、刺激、环境、制品、响应、响应度量`。

## 自检清单

输出前至少核对一次：

1. 每道题是否有明确业务背景，而不是裸知识点。
2. 每个小问是否能对应至少一个稳定采分点。
3. 小问动词是否像真题，而不是像讲义标题。
4. 题目是否覆盖“识别/判断 + 原理/比较 + 设计/优化”中的至少两类。
5. 标准答案是否足够具体，能拿来直接对照评分。
6. 是否已经在 `软考/输出/测试/` 下生成一对独立文件，而不是只生成题目或把答案混写进去。
