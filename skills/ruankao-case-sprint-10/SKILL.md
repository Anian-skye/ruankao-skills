# ruankao-case-sprint-10

这个 Skill 用于把用户上传的案例教材和真题解析变成一个可持续复习的口头抽查流程，而不是一次性的问答。

## 核心目标

1. 每轮默认生成 10 题
2. 从 15 个章节里随机抽 10 个不同章节，每章 1 个知识点
3. 优先抽“高频、重点、超级重点”知识点
4. 回答后记录知识点掌握状态
5. 用户答错时，按 `ruankao-wrong-note` Skill 的格式落到错题本
6. 首轮一次答对且用户没有特别要求保留错题时，将该知识点退休，后续默认不再问
7. 出错过的知识点继续保留在随机池里，后续仍可抽到

## 依赖路径

- 教材 PDF：`软考/资料/案例题/案例教材.pdf`
- 真题 PDF：`软考/资料/案例题/案例真题.pdf`
- 题库：`软考/输出/案例冲刺/题库.json`
- 真题索引：`软考/输出/案例冲刺/真题索引.json`
- 掌握状态：`软考/输出/案例冲刺/掌握情况.json`
- 可视化面板：`软考/输出/案例冲刺/掌握情况.md`
- 调度脚本：`./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py`

## 使用方式

### 1. 先刷新题库和状态

第一次使用，或教材 PDF 更新后，先确认用户已经上传并重命名：

- `软考/资料/案例题/案例教材.pdf`
- `软考/资料/案例题/案例真题.pdf`

然后运行：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py refresh --vault /path/to/vault
```

这一步会：

- 从 PDF 目录页抽取 15 个章节和候选知识点
- 从用户上传的案例真题 PDF 抽取真题索引
- 生成 `题库.json`
- 生成 `真题索引.json`
- 初始化或补齐 `掌握情况.json`
- 重写 `掌握情况.md`

### 2. 生成本轮 10 题

默认抽题命令：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py pick --vault /path/to/vault
```

如需调整联动真题数量，可加：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py pick \
  --vault /path/to/vault \
  --real-count 2
```

要求：

- 默认随机抽取 10 个不同章节，每章 1 题，不重复
- 同一天内优先跳过已经抽过的知识点；只有候选不足时才允许回退到当天已抽题
- 3 天内同一道题（同一 `topic_id + prompt_id`）不能重复；如果旧记录里还没有 `prompt_id`，则对该知识点按 3 天整段冷却处理，避免误复用旧题
- 优先抽未掌握、未退休、重点等级高的知识点
- 已经“首轮一次答对且退休”的知识点不再进入候选池
- 曾答错的知识点即使后续答对，也保留在随机池
- 在抽完 10 道知识点题后，必须从用户上传的案例真题索引中找出至少 2 道和本轮知识点最相关的往年真题，一并提问

如果用户明确要求减少题量、只练指定章节或强制重抽，再按用户要求覆盖默认值。

### 3. 提问方式

默认一次性给出 10 个题目，按章节分组，适合用户统一作答。

在 10 道知识点题之后，默认再追加一个 `关联真题` 区块：

- 至少 2 道最相关往年真题
- 每道真题给出年份、题号、页码
- 给出题干背景摘要
- 列出该真题的 `问题（1）（2）（3）`
- 明确标注它和本轮哪些知识点最相关
- 如需展示真题原页，可把关联真题对应 PDF 页渲染成截图并直接发送图片，优先保留原题图表、空白、选项和版面；截图默认保存到 `~/case-drill-screenshots/real-cases/YYYY-MM-DD/`

每题默认优先模拟历年案例真题的小问风格，尽量使用 `（1）（2）（3）` 这种分问、补全、比较、判断、对应关系的问法，而不是宽泛的“解释一下某概念”。如果一个知识点下面有多个采分点，默认直接抽其中 2 到 3 个采分点来问，不要用“常见考法”“判断关键词”“容易漏写的点”这类元描述代替真正的问题。例如：

- `（1）写出用例图的 3 个基本元素。（2）指出参与者识别时要满足的条件。（3）按顺序写出用例建模步骤。`
- `（1）指出 CAP 中 A 的含义。（2）说明为什么返回旧数据也可能算可用。（3）比较可用性与一致性。`
- `（1）分别写出主从复制、哨兵、Cluster 各解决什么问题。（2）三者里哪一个同时处理分片和高可用。（3）如果题目要求槽位分配和节点扩缩容，对应的是哪一个。`

提问风格必须尽量贴近教材里的历年真题和典型例题，避免过宽的开放题，也避免把“你要考什么”直接说出来。想考哪个采分点，就直接问那个采分点本身。

优先使用下面这几类问法：

- `分别写出 / 指出 / 说明`
- `补全（1）（2）（3）小问`
- `包含哪些角色 / 要素 / 阶段`
- `请从 A、B、C 三个方面比较`
- `补全角色职责 / 步骤顺序 / 关系类型 / 约束条件`
- `给定一个具体业务场景，判断该用哪一种方案并说明原因`

不推荐的宽泛问法：

- `请你口述核心概念`
- `请你说明关键要点`
- `请你谈谈常见考法`
- `指出题干里判断它的关键词`
- `指出一个最容易漏写的点`
- `说明它适用的场景`

只有在题目本身就是大综述型知识点时，才允许用较宽的问法；否则默认拆成 2 到 3 个具体小问。

另外，默认应为同一知识点维护多套更细的题型变体，例如：

- `核心要素 / 角色 / 组成`
- `步骤顺序 / 关键动作 / 约束条件`
- `关系类型 / 组件协作 / 易混概念对比`

也就是说，不要总是重复“定义 + 作用 + 场景”这一套，也不要出现“常见考点 / 关键词 / 易漏点 / 得分点”这种提示式表述；要尽量往更细的采分点拆，以提高有效题库量，同时真的检验用户会不会漏答。

默认优先抽叶子知识点；如果某个目录节点下面还有更细的小节，优先考这些更细的小节，避免总是落到“概述 / 简介 / 大类总结”这种过粗的问题上。

### 3.1 用户作答后的反馈方式

每次用户回答后，默认必须逐题给出以下反馈：

1. `评价`
   - 判定为 `correct / partial / wrong`
   - 简要说明这题答得好的地方和失分点
2. `掌握情况`
   - 明确告诉用户这题目前属于“已掌握 / 有印象但不稳 / 需要重点补洞”中的哪一类
3. `标准答案详解`
   - 直接给出教材口径下的标准答案
   - 尽量按采分点展开，而不是只给一句结论
   - 需要时补充 `为什么这样答`、`和什么概念容易混淆`、`考试里通常怎么给分`

默认不要只报一个总分或一句“对/错”就结束；应让用户能看完就知道：

- 自己这一题错在哪里
- 标准答案应该怎么组织
- 下次遇到相似题该怎么答

如果用户要求“一题一题问”，再改成单题模式。

### 4. 评分与掌握度规则

每题只用三档判断即可：

- `correct`：主干正确，关键点覆盖到位
- `partial`：方向对，但遗漏采分点或表述不稳
- `wrong`：不会、明显答偏、核心概念混淆

掌握度记录约定：

- `0`：空白/答错
- `1`：有印象但不稳定
- `2`：答对过，但不是首轮一次过，仍可继续抽到
- `3`：首轮一次答对，默认退休

### 5. 回写掌握状态

每题判定后立刻回写：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py record \
  --vault /path/to/vault \
  --topic-id c01-1_1_1 \
  --result correct \
  --today YYYY-MM-DD \
  --prompt-id q02 \
  --prompt-text "（1）..." \
  --user-answer "用户原答案" \
  --note "主干正确，但遗漏响应度量"
```

默认行为：

- `correct` 且这是第一次被问到：标记为 `retired`
- `partial`：保留在池中
- `wrong`：保留在池中，后续仍可抽到

如果用户明确说“即使答对也继续反复问”或“这题也要记错题”，再加：

```bash
--force-keep
```

### 6. 判题前如何回看教材

如果需要依据教材判断标准答案，不要凭印象硬判。先读取该知识点原文：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py show \
  --vault /path/to/vault \
  --topic-id c01-1_1_1
```

这会根据目录中的页码区间抽出该小节的 PDF 文本，供你快速整理：

- 标准答案
- 采分点
- 易混点

如果需要回看某道关联真题原文，使用：

```bash
python3 ./.agents/skills/ruankao-case-sprint-10/scripts/case_drill.py show-real \
  --vault /path/to/vault \
  --real-question-id rq-2021-11-2
```

## 与错题 Skill 的联动

当用户某题判定为 `wrong` 时，必须按 `ruankao-wrong-note` Skill 的格式记录错题。

至少要组织出下面这些信息：

```markdown
## 错题信息
- 题目：本轮抽查题目
- 我的答案：用户原答案
- 正确答案：根据教材整理出的标准答案
- 错误原因：知识点遗漏 / 理解偏差 / 审题不清
- 知识点：对应知识点名称
```

然后按 `ruankao-wrong-note` Skill 的规则：

- 写入当天 `软考/输出/错题本/YYYY-MM-DD.md`
- 与 `软考/输出/知识点索引.md` 建立双向链接

## 选择知识点的原则

优先级从高到低：

1. `超级重点★★★★★`
2. `重点★★★★★`
3. `次重点★★★☆☆`
4. 三级标题优先于二级标题
5. 曾答错过的知识点优先于完全没问过的普通知识点

以下类型默认降权或过滤：

- `小结`
- `典型例题`
- `某某系统案例`
- 纯导读性质的小节

## 输出要求

给用户展示题单时：

- 按章节分组
- 每章 1 题
- 总数默认 10 题
- 保持题目短、适合口头作答

给用户展示结果时：

1. 先给整体表现
2. 再给本轮答错/薄弱知识点
3. 再逐题给出评价与标准答案详解
4. 再说明哪些知识点已退休，不再默认重复
5. 最后提醒下次可直接继续案例冲刺抽查

## 注意事项

- 默认使用中文
- 不要把“答对一次”和“真正稳定掌握”混为一谈；这里只是按用户要求把“首轮一次答对”从默认池中退休
- 如果教材目录解析失败，先提示，再手动从目录页补充 15 章结构，不要中断整个流程
- 如果用户只回答了部分题目，只更新那些已经判定的题，不要虚构其余结果
