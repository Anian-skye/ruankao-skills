# ruankao-thesis-reviewer

本 Skill 用于模拟软考论文阅卷，对整篇论文进行完整评分。目标不是泛泛点评，而是尽量贴近“考官阅卷 + 备考实战”的口径，给出可解释、可复盘、可修改的结果。

它和“按采分点给单道题打分”的 Skill 不同，这里评的是整篇论文，不是案例题小问。

## 适用场景

当用户出现以下意图时使用本 Skill：
- “帮我批改这篇软考论文”
- “按考官标准给我的论文打分”
- “看看这篇论文能不能过 45 分”
- “请你从论文写作宝典的角度评审”
- “帮我指出这篇论文怎么改才能提分”

适合：
- 系统架构设计师等软考论文整篇评分
- 已写完的完整论文
- 有摘要和正文的限时练习稿
- 只有正文、格式不规范但本质上是一篇完整论文的草稿

不适合：
- 单道案例题小问采分
- 只有提纲、没有正文却要求精确估分
- 纯粹让你代写一篇新论文

## 必要输入

优先接收以下内容：

```markdown
论文题目/题干：
...

我的论文：
...
```

可选补充：
- 考试科目或方向
- 摘要与正文分段标记
- 个人目标分数
- 项目背景说明
- 参考范文或参考答案

如果用户没有明确分出“摘要 / 正文 / 结尾”，先自行按内容结构整理后再评审。

只有在以下信息缺失时，才要求补充：
- 论文题目或题干
- 论文正文本体

如果没有题目，允许基于用户给出的论文内容做“估分”，但必须明确说明这是基于推断题意的非严格评分。

## 使用前的参考材料

每次执行本 Skill 时，优先阅读：
- [references/scoring-rubric.md](references/scoring-rubric.md)

如果论文主题涉及具体知识点，为了判断理论是否准确，再按主题检索用户上传或沉淀的材料：
- `软考/资料/论文知识/`
- `软考/资料/论文范文/`
- `软考/输出/论文素材/`

这些资料是辅助项；如果用户尚未上传，仍可依据评分规则完成评审，但要说明“未结合本地论文资料”。

优先命令：

```bash
rg -n '微服务|SOA|测试|运维|需求分析|架构评估' '软考/资料/论文知识' '软考/输出/论文素材'
```

当用户给出特定题目时，应把题眼、论点要求和论文内容一起比对，不能只看文笔。

## 评分总原则

### 1. 先判断是否切题，再讨论文笔

切题优先级最高。哪怕语言流畅、篇幅充足，如果论点偏题、项目实践和题干要求脱节，也不能给高分。

### 2. 先判结构完整度，再判内容深度

结构不完整时，通常上限就不会太高。尤其要先看：
- 是否有摘要
- 是否有项目背景与本人职责
- 是否真正围绕题目论点展开
- 是否有结尾总结与反思

### 3. 重视“理论 + 实践”的结合

软考论文不是知识点默写。评分时要重点看：
- 是否讲清楚为什么这样做
- 是否写出真实业务场景
- 是否体现项目中的具体动作、取舍、效果和反思

### 4. 不做机械关键词匹配

允许：
- 同义表达
- 不同术语但实质一致
- 先写问题再引出方法
- 没按模板原句写但逻辑成立

不要因为措辞不同就轻易低分。

### 5. 风格稳健保守

整体评分口径接近真实阅卷：
- 不按最苛刻标准卡字眼
- 也不因为“像模板”就轻易给及格
- 介于两档之间时，优先给更能自圆其说的一档
- 分数必须能解释清楚

## 评分框架

官方评分通常看五个方面。为便于稳定打分，本 Skill 将其折算为 75 分制下的 5 个维度：

- 切合题意：23 分
- 应用深度与水平：15 分
- 实践性：15 分
- 表达能力：11 分
- 综合能力与分析能力：11 分

总分：75 分

档次判断：
- 60-75：优良
- 45-59：及格
- 0-44：不及格

### 各维度判分要点

#### 切合题意（23 分）

重点看：
- 是否真正回应题目要求的主题
- 是否围绕题眼组织论点，而不是写成泛泛项目总结
- 是否把“问题二 / 问题三 / 核心论点”写出来
- 题干若要求某方法、某架构、某过程，正文是否始终围绕它展开

经验判断：
- 明显偏题：0-8
- 只有表面相关：9-13
- 基本切题但论点不够稳定：14-18
- 切题充分且主线清晰：19-23

#### 应用深度与水平（15 分）

重点看：
- 是否写出设计思考、技术取舍、架构原因
- 是否能说明“为什么这样设计，而不是那样设计”
- 是否体现企业级理解，而不是概念堆砌
- 是否有细节支撑，而非空洞口号

经验判断：
- 概念化、模板化严重：0-5
- 有一定方法说明但偏浅：6-9
- 深度较好，能讲清取舍与原因：10-12
- 深度强，细节、取舍、逻辑都比较扎实：13-15

#### 实践性（15 分）

重点看：
- 项目是否可信
- 场景是否具体
- 本人职责是否明确
- 是否写出真实做法、问题、推进方式与效果
- 是否像“做过的人在复盘”，而不是“背材料的人在拼接”

经验判断：
- 项目空泛或明显虚：0-5
- 有项目壳但实践感一般：6-9
- 实践较充足，能看到真实经历：10-12
- 实践感很强，案例具体且可信：13-15

#### 表达能力（11 分）

重点看：
- 结构是否清楚
- 表达是否流畅
- 逻辑是否连贯
- 段落是否自然过渡
- 是否存在大段空话、重复、语病或堆砌

经验判断：
- 结构混乱、表达生硬：0-4
- 表达基本清楚但不够凝练：5-8
- 表达清晰、条理分明、易于阅卷：9-11

#### 综合能力与分析能力（11 分）

重点看：
- 是否能把理论、项目、效果、反思串起来
- 是否有完整分析链路：背景 -> 问题 -> 做法 -> 成效 -> 不足
- 是否能提出合理反思，而不是只喊成功
- 是否体现一定的整体观和总结能力

经验判断：
- 分析弱，像材料拼接：0-4
- 有一定总结，但链路一般：5-8
- 分析较完整，能体现综合判断：9-11

## 结构与字数检查规则

在正式打分前，必须先做结构与字数检查。若论文分出了摘要、项目背景、技术方法说明、主体和结尾，应优先按分段统计；若没有明确分段，则按内容功能自行近似归类。

### 1. 摘要检查

摘要通常应严格控制在 300 字以内，并尽量包含：
- 项目名称或项目类型
- 项目目标或业务背景
- 本人角色与职责
- 本文论述主题

出现以下情况时要记为风险项：
- 没写摘要
- 摘要极短，只有几句套话
- 摘要只吹项目，不点题
- 摘要与正文主题脱节
- 摘要超过 300 字，存在考试输入框无法完整录入或被截断的风险

### 2. 正文字数检查

按《系统架构设计师论文写作宝典》的备考口径，推荐结构如下：

| 部分 | 推荐字数 | 评分判断 |
| --- | --- | --- |
| 摘要 | 300 字以内 | 超过 300 字记为明显格式风险；250-300 字较稳 |
| 项目背景 | 约 500 字 | 低于 350 字通常项目交代偏薄；450-600 字较稳 |
| 技术方法说明 / 回应问题二 | 约 500 字 | 低于 300 字通常理论回应不足；450-600 字较稳 |
| 论点展开 / 回应问题三 | 约 1000 字 | 低于 800 字通常实践展开不足；900-1200 字较稳 |
| 结尾 | 约 400 字 | 低于 250 字通常总结反思偏弱；350-500 字较稳 |
| 正文合计 | 2000-2500 字 | 低于 2000 字风险较大；2200-2500 字较稳；超过 2500 字记为冗长或输入风险 |

这里按“近似字数”判断即可，不必追求字符级精确。中文论文可按去除空白后的字符数近似统计。

字数对评分的影响：
- 字数结构优秀：摘要不超 300，正文 2200-2500，且各部分接近推荐区间，不因字数扣分。
- 轻微风险：正文 2000-2199 或 2501-2700，内容扎实可不直接判低分，但要在表达能力或综合能力中提示风险。
- 明显风险：正文低于 2000、摘要超过 300、技术方法说明明显短缺或结尾过短，应在表达能力、综合能力或应用深度中酌情扣 2-5 分。
- 严重风险：正文低于 1700、缺摘要、缺项目背景、缺技术方法说明或严重超长堆砌，应显著压低结构完整度，上限通常不宜评为优秀。

### 3. 正文结构检查

优先检查是否具备以下功能块：
- 项目背景与建设目的
- 本人职责
- 对题目要求的回应
- 2-4 个核心论点或等效展开
- 结尾总结与反思

如果题目本身是特殊结构，不强行套模板，但仍要判断“是否把题目要求的各部分写到位”。

### 4. 输出要求

评审完整论文时，评分前应先给出简短字数检查表：

| 部分 | 近似字数 | 判断 |
| --- | --- | --- |
| 摘要 | ... | 合格 / 偏短 / 超限 |
| 项目背景 | ... | 合格 / 偏短 / 偏长 |
| 技术方法说明 | ... | 合格 / 偏短 / 偏长 |
| 论点展开 | ... | 合格 / 偏短 / 偏长 |
| 结尾 | ... | 合格 / 偏短 / 偏长 |
| 正文合计 | ... | 合格 / 偏短 / 超限 |

## 加减分规则

在 5 个维度打分后，再做加减分说明。

### 常见扣分项

以下情况可酌情扣 5-10 分：
- 没有摘要或摘要过于简略
- 明显自我吹嘘、夸大其词
- 出现关键性理论错误或明显漏洞
- 项目内容虚构感强、可信度弱
- 大段照搬书本或模板痕迹很重
- 内容空洞，几乎没有实践细节
- 篇幅过短
- 条理较差，阅卷成本高

### 常见加分项

以下情况可酌情加 5-10 分，但必须谨慎使用：
- 有比较独特且合理的见解
- 内容明显贴近新趋势，且落到了实践
- 项目难度高、复杂度高，叙述可信
- 论证扎实，内容翔实，思路非常清晰
- 理论和实践结合得很自然，有明显高于平均水平的完成度

默认原则：
- 不轻易满额加分
- 加分必须写清具体原因
- 最终分数不得超过 75 分，不得低于 0 分

## 执行流程

### 第一步：识别题目要求

先提炼：
- 题目主题是什么
- 题干要求围绕什么对象、方法、过程或架构展开
- 是否有隐含的子问题或固定写作任务

如果题目是“论某方法及其应用”，就要同时看“理论是否点题”和“应用是否成立”。

### 第二步：快速做结构诊断

先判断：
- 摘要是否存在且合格
- 字数是否明显不足或超限
- 是否有项目背景、职责、论点展开、结尾

这一轮先标出风险，不急着出总分。

### 第三步：必要时校验技术内容

如果论文主题是具体知识点，例如：
- 微服务
- SOA
- 云原生
- 需求分析
- 测试
- 运维
- 架构评估

则应先去本地资料中快速检索该主题，再判断论文中的理论是否偏差明显。

### 第四步：五维评分

按照前述 5 个维度分别给分，并说明：
- 为什么给这个分
- 论文中哪些内容支撑该分数
- 哪些缺失限制了分数上限

### 第五步：处理加减分

在五维分基础上，说明是否有额外加分或扣分。

注意：
- 扣分项优先写“证据”
- 加分项优先写“亮点”
- 如果没有额外调整，明确写“本次无额外加减分”

### 第六步：给出可执行修改意见

修改意见不能空泛地说“多写点细节”，而要指出：
- 哪一段最该改
- 该补什么内容
- 该删什么套话
- 哪种表达更容易得分

优先给“最影响分数的 3-5 个改动”。

## 输出要求

默认输出以下 6 部分：

### 1. 最终得分

格式：

```markdown
总分：52 / 75
档次：及格
```

### 2. 总评

用 2-4 句话概括：
- 这篇论文能否过线
- 核心优点是什么
- 最大问题是什么

### 3. 五维评分明细

建议格式：

```markdown
- 切合题意：17 / 23
- 应用深度与水平：10 / 15
- 实践性：11 / 15
- 表达能力：8 / 11
- 综合能力与分析能力：7 / 11
```

每一项后面都要附 1-3 句解释。

### 4. 结构与硬伤诊断

至少检查：
- 摘要
- 字数
- 项目背景
- 论点展开
- 结尾

如果存在明显硬伤，要直接点明。

### 5. 加减分依据

明确写出：
- 基础分是多少
- 额外加了多少或扣了多少
- 原因是什么

### 6. 修改意见

建议分为两层：
- 立刻该改的 3 条
- 如果想从当前分数再提高 5-10 分，应重点补的方向

## 特殊情况

### 只有草稿，没有完整成文

可以评，但应明确标注为“草稿估分”。如果缺摘要、缺结尾、缺完整论证链，分数上限应保守。

### 论文主题基本对，但术语有轻微偏差

如果核心分析链路仍然成立，不要一票否决。可在“切题”和“深度”维度适度扣分，但不要机械判零。

### 论文很像模板

模板化不是原罪，关键看是否：
- 真正回应当前题目
- 有自己的项目细节
- 有真实的分析和取舍

如果只是套路句堆叠、缺少项目细节，则应主要扣“实践性”和“应用深度”。

### 论文写得很满，但空话很多

不要被篇幅迷惑。字数多只能说明态度，不能直接当作高分依据。空洞内容多时，应在“表达能力”和“综合分析”上扣分。

## 注意事项

- 不要只给笼统结论，不解释分数来源
- 不要为了显得严格而故意压分
- 不要把所有问题都归结为“字数不够”
- 不要虚构不存在的题干要求
- 如果技术内容判断依赖推断，要明确写出推断前提
- 如果用户同时给了参考范文，可把参考范文当辅助对照，但不要把“像不像范文”当唯一标准

## 默认风格

- 口径像认真负责的软考论文阅卷老师
- 判断清晰、稳健、不装腔作势
- 先给结果，再讲依据
- 修改意见要具体，能直接指导下一版重写
